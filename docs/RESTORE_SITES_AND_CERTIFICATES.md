# Срочное восстановление сайтов, доменов и сертификатов на сервере

**Важно:** Ничего не затирать и не удалять. Только восстановить привязки и доступность.

Все команды ниже выполняются **на сервере** (SSH).

---

## 1. Восстановление из бэкапа (если есть)

Если перед изменениями делали бэкап конфигов и сертификатов:

```bash
# Пример: восстановление nginx (пути подставьте свои)
sudo cp -a /backup/nginx/sites-available/* /etc/nginx/sites-available/
sudo cp -a /backup/nginx/sites-enabled/*   /etc/nginx/sites-enabled/

# Пример: восстановление Let's Encrypt (если бэкапили)
sudo cp -a /backup/letsencrypt/* /etc/letsencrypt/

sudo nginx -t && sudo systemctl reload nginx
```

Если бэкапа нет — переходите к шагу 2.

---

## 2. Текущее состояние: что есть на сервере

### 2.1 Сайты Nginx

```bash
# Все конфиги в sites-available
ls -la /etc/nginx/sites-available/

# Что реально включено (симлинки в sites-enabled)
ls -la /etc/nginx/sites-enabled/

# Проверка синтаксиса
sudo nginx -t
```

### 2.2 Сертификаты Let's Encrypt

```bash
# Список всех выданных сертификатов
sudo certbot certificates
```

По выводу видно, какие домены привязаны к каким сертификатам и где лежат ключи.

---

## 3. Восстановить привязки сайтов (не удаляя существующие)

- **Не удалять** файлы из `sites-available` и не очищать `sites-enabled`.
- Если какой-то сайт пропал из `sites-enabled` — вернуть его симлинком.

```bash
cd /etc/nginx/sites-enabled

# Пример: включить сайт, если он есть в sites-available, но отключен
# (имя файла подставьте по факту из sites-available)
sudo ln -sf ../sites-available/имя-сайта.conf .

# Проверить
ls -la
sudo nginx -t
sudo systemctl reload nginx
```

Повторить для каждого недостающего домена/поддомена (файл из `sites-available` → симлинк в `sites-enabled`).

---

## 4. Восстановить сертификаты (только недостающие)

- **Не запускать** `certbot --nginx` по всем доменам — это может переписать конфиги.
- Выдавать сертификаты **только для конкретного домена**, если его нет в `certbot certificates`.

### 4.1 Выдать сертификат только для одного домена (без изменения nginx)

```bash
# Только для admin.neeklo.ru (пример). Остальные домены не трогать.
sudo certbot certonly --webroot -w /var/www/html -d admin.neeklo.ru
```

Для других доменов/поддоменов — свои `-d` и при необходимости свой `-w` (путь к корню для ACME challenge).

### 4.2 Если конфиг сайта уже есть и в нём указаны пути к сертификатам

Убедитесь, что в конфиге Nginx для этого домена указаны правильные пути, например:

```nginx
ssl_certificate     /etc/letsencrypt/live/ДОМЕН/fullchain.pem;
ssl_certificate_key /etc/letsencrypt/live/ДОМЕН/privkey.pem;
```

После выдачи серта путь будет `/etc/letsencrypt/live/ДОМЕН/`. Перезагрузка Nginx:

```bash
sudo nginx -t && sudo systemctl reload nginx
```

---

## 5. Если конфиг сайта был перезаписан

Если для какого-то домена конфиг был заменён одним файлом (например, только admin.neeklo.ru):

1. Восстановить старый конфиг из бэкапа (см. п. 1), **или**
2. Вручную собрать конфиг заново: один `server { }` на домен/поддомен, с правильными `server_name`, `root`, `ssl_certificate*` и т.д.
3. Файлы из репозитория (например, `tools/deploy/admin.neeklo.ru.nginx.conf`) — **только образец для одного домена**. Не копировать их поверх всех сайтов: использовать как образец и **добавлять** в конфигурацию, не заменяя другие сайты.

---

## 6. Чек-лист после восстановления

- [ ] `sudo nginx -t` — без ошибок
- [ ] `sudo systemctl status nginx` — active (running)
- [ ] `sudo certbot certificates` — все нужные домены есть
- [ ] Проверить в браузере или curl: каждый домен/поддомен открывается по HTTP/HTTPS как задумано
- [ ] Для admin.neeklo.ru: проверка `https://admin.neeklo.ru/crm` и при необходимости webhook бота

---

## 7. На будущее: что не делать

- **Не выполнять** `certbot --nginx` без указания конкретного домена и без понимания, что он может менять конфиги.
- **Не копировать** один конфиг Nginx поверх всей папки `sites-available` или `sites-enabled`.
- **Не удалять** и не затирать существующие конфиги других сайтов при добавлении нового (admin.neeklo.ru или иного).
- Новый сайт добавлять отдельным файлом в `sites-available` и симлинком в `sites-enabled`, не трогая остальные.

Конфиг для admin.neeklo.ru в репозитории — **образец**. На сервере его нужно **добавить** к уже существующим сайтам, а не подменять ими все виртуальные хосты.
